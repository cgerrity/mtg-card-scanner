# Python Dependency Management

**TL;DR:** Use `pyproject.toml` for project metadata and tool config; use `requirements.txt` (or a lockfile) for the actual dependency versions. Pin to **at least the major version** of every direct dep. Separate runtime and dev dependencies.

---

## What it is

Three artifacts you encounter in modern Python projects, often confused:

### `pyproject.toml`

The **modern standard** for Python project config. Two flavors:

1. **Library mode** — includes a `[build-system]` section and a build backend (hatchling, setuptools, poetry). Used when you intend `pip install -e .` or publishing to PyPI.
2. **Script project mode** — only has `[tool.*]` sections (e.g. `[tool.pytest.ini_options]`, `[tool.ruff]`). No build system. Used for "I have scripts and shared code but I don't publish."

This project uses flavor 2.

### `requirements.txt`

A flat list of dependencies, one per line, optionally with version specifiers:

```
requests>=2.32.0,<3.0
ijson>=3.3.0,<4.0
```

Read by `pip install -r requirements.txt`. Older format, but still the simplest way to express "install these for me."

### Lockfile (e.g. `requirements.lock`, `poetry.lock`, `uv.lock`)

A **fully resolved** list of every transitive dependency at exact versions. Generated by a tool, not hand-edited.

```
requests==2.32.3
charset-normalizer==3.3.2
idna==3.7
urllib3==2.2.2
certifi==2024.2.2
```

This is what production deploys should pin to. `pip freeze` gives you a quick-and-dirty version; `pip-tools`, `uv`, or `poetry` give a more robust one.

## Why it matters

For ML eng work, dependency hell is real:

- **PyTorch + CUDA version drift** — pip-installing `torch` blind can give you the wrong CUDA build for your GPU.
- **NumPy ABI breakage** — major version changes break compiled extensions.
- **Transitive conflicts** — your dep wants `protobuf<4`, another dep wants `protobuf>=5`. Resolver fails.
- **"Works on my machine"** — the classic. A teammate has a slightly different transitive dep tree.

A senior interview question often is: **"You hand me your training script. I run it and it explodes. Walk me through how to make this not happen."** Answer: pinning, lockfiles, container the environment.

## Pinning strategies

| Strategy | When | Example |
|---|---|---|
| Unpinned | Never | `requests` |
| Caret (npm-style) | When you trust the dep's semver discipline | `requests>=2,<3` |
| Tilde | Patch only | `requests>=2.32,<2.33` |
| Exact pin | Production deploys, reproducibility | `requests==2.32.3` |
| Hash pin | High-security environments | `requests==2.32.3 --hash=sha256:abc...` |

**Our project's strategy:** caret-style in `requirements.txt` (`>=X.Y,<X+1`) for direct deps. We trust major-version pins. When we reach Phase 6 we'll add a lockfile.

## Runtime vs dev split

Why we have `requirements.txt` + `requirements-dev.txt`:

```
requirements.txt              ← what production needs to RUN
requirements-dev.txt          ← above + what developers need to TEST/LINT
```

The dev file uses `-r requirements.txt` to inherit the runtime set:

```text
# requirements-dev.txt
-r requirements.txt
pytest>=8.0.0,<9.0
ruff>=0.5.0,<1.0
```

**Why:** production should never have pytest installed. It's attack surface, takes space, and a CI/CD pipeline that conflates "install dev tools" with "install runtime" is a code smell. The same instinct applies to ML serving — your inference container should be lean (just torch + your model code), not the full training environment.

## Modern alternatives — what to know

Going forward in ML eng circles you'll see:

- **`uv`** — by Astral (the ruff people). Fast (Rust). Combines venv + pip + lock generation. Increasingly popular as the modern default.
- **`poetry`** — full-featured project manager. More opinions; bigger surface area.
- **`pdm`** — PEP 582-aligned; targets the same niche as poetry.
- **`pipenv`** — older attempt; deprecated in practice.
- **`conda`** / **`mamba`** — packages C/C++ binaries (CUDA, MKL). Often required for ML libraries with system deps. Slower than pip.

For this project we stay with pip + venv (the lowest-friction option for a learning project). When we hit Phase 6 and need PyTorch with CUDA, we may reach for conda.

## Watch out for

- **`pip install <pkg>` without checking how it lands.** Always `pip list` after to verify the version you got.
- **Mixing pip and conda in the same env.** Can corrupt the env. Pick one per env.
- **Forgetting `-r` in `requirements-dev.txt`.** Without it you only install dev deps, not runtime.
- **Lockfiles in CI but not local.** Your CI is reproducible; your local isn't. Misery.
- **Pinning so loose that `requests>=2` allows `requests==3.0.0` someday.** The `<3` upper bound matters.

## Interview angle

> **"How do you manage Python dependencies in your projects?"**

A senior answer covers:

1. `pyproject.toml` for metadata and tool config
2. Separate runtime vs dev dependencies
3. Pinning policy (caret for direct deps, exact for production)
4. Lockfile for production deploys
5. Tool choice rationale (pip vs uv vs conda) and when to escalate to conda for system-level deps
6. CI checks: dependency resolution succeeds, no security advisories (e.g. `pip-audit`)

Bonus points: mention that for ML serving you build minimal inference containers (no dev/training deps).
